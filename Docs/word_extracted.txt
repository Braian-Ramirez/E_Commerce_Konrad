[Title] Documento de Arquitectura de Software
[Normal] Visión General de la Arquitectura
[Normal] La arquitectura del sistema E-Commerce Konrad se basa en un enfoque de Microservicios con comunicación mediante API Gateway, siguiendo principios de Domain-Driven Design (DDD). Esta decisión responde a los requerimientos de alta disponibilidad, escalabilidad a 200.000 usuarios concurrentes, computación distribuida y facilidad de mantenimiento independiente de cada módulo.
[Normal] Patrones y Estilos Arquitectónicos Seleccionados
[Normal] Microservicios — Patrón Principal
[Normal] Justificación: Cada dominio funcional (vendedores, compradores, pagos, catálogo, notificaciones, auditoría) se implementa como un servicio independiente con su propia base de datos. Esto permite escalar, desplegar y mantener cada módulo de forma autónoma, lo cual es crítico para soportar 1.000 TPS en el módulo de pagos sin afectar el módulo de catálogo.
[Normal] Servicios identificados: Auth Service, Vendor Service, Buyer Service, Product Service, Order Service, Payment Service, Notification Service, Audit Service, BAM Service, File Processor Service
[Normal] API Gateway — Patrón de Acceso Unificado
[Normal] Justificación: Actúa como punto de entrada único para todos los clientes (web, mobile). Gestiona autenticación, enrutamiento, rate limiting, SSL termination y logging. Se propone el uso de Kong o AWS API Gateway (open source friendly).
[Normal] Event-Driven Architecture (EDA) — Patrón de Integración Asíncrona
[Normal] Justificación: Requerimientos como el procesamiento de archivos planos de CIFIN y del banco (consignaciones), el envío de correos, la detección de tendencias y el cambio de estado de suscripciones son procesos asincrónicos. Se usa Apache Kafka o RabbitMQ como broker de eventos para desacoplar estos procesos.
[Normal] Ejemplo: cuando el banco deja el archivo de consignaciones, el File Processor Service publica un evento 'ConsignacionDetectada', el Payment Service lo consume y actualiza el estado de la orden.
[Normal] Repository Pattern — Patrón de Acceso a Datos
[Normal] Justificación: Requerimiento RNF-14 exige que el sistema esté preparado para cambiar el motor de base de datos. El patrón Repositorio abstrae la lógica de persistencia del dominio, permitiendo sustituir PostgreSQL por otro motor sin afectar la lógica de negocio.
[Normal] Adapter Pattern — Integración con Sistemas Externos (Mocks)
[Normal] Justificación: Los sistemas externos (Datacrédito REST, CIFIN archivo plano, Policía portal web, pasarela de pagos, banco) tienen interfaces heterogéneas. El patrón Adapter (también llamado Wrapper) permite encapsular cada integración en un módulo intercambiable. En el proyecto se implementarán como Mocks que simulan las respuestas de cada servicio externo, lo que facilita las pruebas sin depender de infraestructura real.
[Normal] DatacreditoAdapter → Mock REST que retorna: Baja / Alta / Advertencia
[Normal] CifinAdapter → Mock que lee archivo plano y retorna: Baja / Alta / Advertencia
[Normal] PoliceAdapter → Mock que retorna: Requerido / No requerido
[Normal] PaymentGatewayAdapter → Mock que simula pago en línea y tarjeta de crédito
[Normal] BankFileAdapter → Mock que genera archivo plano de consignaciones diarias
[Normal] CQRS (Command Query Responsibility Segregation) — Patrón para Reportes y BAM
[Normal] Justificación: El tablero de control (BAM) con KPIs y la consulta de auditoría requieren lecturas de alto rendimiento que no deben competir con las escrituras transaccionales. CQRS separa los modelos de escritura y lectura, optimizando cada uno de forma independiente.
[Normal] Facade Pattern — Simplificación de Subsistemas Complejos
[Normal] Justificación: El proceso de revisión de una solicitud de vendedor involucra múltiples subsistemas (Datacrédito, CIFIN, Policía). El patrón Facade expone una interfaz unificada 'EvaluarSolicitud' que internamente orquesta las tres consultas y retorna un resultado consolidado al Director Comercial.
[Normal] Observer Pattern — Notificaciones y Automatización
[Normal] Justificación: Cambios de estado en solicitudes, vencimientos de suscripción, calificaciones bajas de vendedores y detección de tendencias activan notificaciones automáticas por correo. El patrón Observer desacopla la lógica de negocio del mecanismo de notificación.
[Normal] Strategy Pattern — Módulo de Pagos
[Normal] Justificación: El sistema acepta tres formas de pago (en línea, tarjeta de crédito, consignación). El patrón Strategy permite encapsular cada algoritmo de pago en una clase independiente e intercambiarla en tiempo de ejecución según la elección del usuario.

[Title] Plan de Pruebas
[Normal] Tipos de Pruebas
[Normal] Pruebas Unitarias: Cada microservicio tiene cobertura de pruebas unitarias sobre la lógica de negocio (mínimo 80% de cobertura). Herramienta: Jest (Node.js) / JUnit (Spring Boot).
[Normal] Pruebas de Integración: Validación de la comunicación entre microservicios y los mocks de sistemas externos. Herramienta: Postman / Newman.
[Normal] Pruebas de Sistema: Validación end-to-end de los flujos completos del negocio.
[Normal] Pruebas de Carga: Validación del RNF-06 (200.000 usuarios concurrentes, 1.000 TPS). Herramienta: Apache JMeter.
[Normal] Pruebas de Seguridad: Verificación de HTTPS, encriptación de contraseñas, control de acceso por roles. Herramienta: OWASP ZAP.
[Normal]  Casos de Prueba Funcionales
[Normal]  Criterios de Aceptación Generales
[Normal] Todos los flujos funcionales del diagrama de casos de uso deben pasar sus pruebas de sistema.
[Normal] La cobertura de pruebas unitarias debe ser superior al 80% en todos los microservicios.
[Normal] Las pruebas de seguridad no deben revelar vulnerabilidades críticas (OWASP Top 10).
[Normal] Las pruebas de carga deben confirmar la capacidad de 200.000 usuarios concurrentes.
[Normal] Todos los mocks deben estar correctamente documentados y ser intercambiables con la implementación real mediante configuración.

--- Tabla 3 (7 filas x 2 columnas) ---
ID | CU-01
Nombre | Registrar solicitud de vendedor
Actor Principal | Solicitante (persona natural o jurídica)
Precondición | El solicitante accede a la página de E-Commerce Konrad sin autenticación previa
Flujo Principal | 1. El sistema presenta el formulario de registro. 2. El solicitante ingresa: nombres, apellidos, número de identificación, correo, país, ciudad, teléfono. 3. El sistema valida el formato de los datos. 4. El sistema determina si es persona natural o jurídica y solicita los documentos requeridos. 5. El solicitante adjunta los documentos (cédula, RUT, cámara de comercio si aplica, formatos de aceptación). 6. El sistema valida documentos y registra la solicitud en estado PENDIENTE. 7. El sistema muestra el número de solicitud generado.
Flujo Alternativo | 3a. Si los datos tienen formato inválido, el sistema indica el error y solicita corrección.
Postcondición | Solicitud registrada en estado PENDIENTE. Solicitante recibe el número de radicado.

--- Tabla 4 (6 filas x 2 columnas) ---
ID | CU-07
Nombre | Aprobar solicitud de vendedor
Actor Principal | Director Comercial
Precondición | El Director ha consultado y revisado la solicitud. Las calificaciones crediticias son ALTA en CIFIN y Datacrédito. No tiene requerimiento judicial.
Flujo Principal | 1. El Director registra resultado APROBADA. 2. El sistema actualiza el estado de la solicitud. 3. El sistema genera credenciales de acceso. 4. El sistema envía correo certificado con credenciales al solicitante.
Postcondición | Solicitud en estado APROBADA. Vendedor puede ingresar a la plataforma y seleccionar plan de suscripción.
